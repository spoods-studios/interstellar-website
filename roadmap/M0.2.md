# Roadmap Detail — M0.2 Coordinate System

> **Granular retroactive roadmap.** One entry per execution phase —
> the phase-level detail layer beneath the milestone-level roadmap.

**Milestone:** M0.2 Coordinate System — three-layer coordinates: int64 mm
storage (exact), float64 relative physics, float32 camera-relative
rendering. Inertial frame only. First milestone closed through a review
gate.
**Era:** Era 0 (engine foundations).
**Phases:** 6 — numbered 7–11 plus inserted 10.5 (global numbering
continues from M0.1's 1–6).

## Phase 7: Core Vector Types — ✅ 2026-04-22

**Goal:** three typed Vec3 structs, one per coordinate layer, with complete
math operations and unit tests. 23 decisions were locked in this
phase, executed with zero deviations from plan.

**What it delivered:**
- `Vec3i64` (int64 storage, 1 mm/unit, ±600 AU range), `Vec3f64` (float64 relative physics), `Vec3f32` (float32 camera-relative rendering) — three separate structs in `interstellar::coords`, header-only
- No inheritance, no template, no mixed-type operators: layer crossings are compile errors, pinned by `static_assert` tests on all six conversion directions
- `Vec3i64` restricted to integer-exact operations (`dot`, `distance_squared`, spaceship ordering) — no `length`/`normalize`, which would need floating point on the exact layer
- 32 unit tests: bit-exact for integer ops, `Approx` margins for float ops; first real test suite in the repo (replaced M0.1's placeholder)

**Key decisions:** precision-encoding names (`Vec3f64` serves position and
velocity; role names rejected); `Vec3<T>` template rejected — separate
structs make implicit-conversion prevention free; free-function conversion
style fixed for Phase 8 before conversions existed; module folder
(`types.hpp` / `conversions.hpp` / `service.hpp`) laid out up front.

**Deep-dive:** posted in #technical-devlog

## Phase 8: Layer Conversions — ✅ 2026-04-22

**Goal:** explicit conversion functions between the three coordinate layers
— the only way data crosses layer boundaries.

**What it delivered:**
- Four `[[nodiscard]] inline` free functions in `conversions.hpp`: `to_relative` (storage→physics), `to_camera_relative` (storage→rendering, 4 km debug assert), `to_absolute` (physics→storage, round + origin), `to_render` (physics→rendering narrowing)
- Subtract-before-convert enforced structurally: int64 subtraction first (exact), float cast second — catastrophic cancellation prevented by function shape
- Proof test: positions at 2⁵⁴ mm with 3 mm delta — float64 ULP there is 4 mm, so convert-first loses the delta entirely, subtract-first returns it exactly
- Round-trip contract tested to 1 AU: `to_absolute(to_relative(pos, origin), origin) == pos` bit-exact within 2⁵³
- `mm_to_meters` / `meters_to_mm` named scale constants

**Key decisions:** precision rules as executable code (function structure +
asserts), not documentation; free functions so layer crossings are greppable
call sites.

**Deviations:** none in-phase — the heaviest post-phase revision in the
module came later, across 11 follow-up commits: replaced `llround`
with `nearbyint` (half-to-even; away-from-zero rounding biases
round-trip chains), replaced an inexact `×0.001f` with exact
`÷1000.0f`, made contract violations Release-observable, and M0.4
hoisted the range guard above a UB float→int cast.

**Deep-dive:** posted in #technical-devlog

## Phase 9: Coordinate Service — ✅ 2026-04-22

**Goal:** a stateful service owning the current origin — the single value
physics and rendering both query, per the 2026-04-21 single-source-of-truth
decision.

**What it delivered:**
- `CoordinateService` in `service.hpp` / `coordinate_service.cpp`: `Vec3i64` origin plus configurable shift threshold (default 1 km = 1,000,000 mm), non-copyable/non-movable per the VulkanContext pattern
- Six public methods: `get_origin`, `shift_origin`, `should_shift`, `distance_from_origin`, `to_relative`, `to_camera_relative` — the last two one-line delegations to the Phase 8 free functions
- `should_shift` compares squared distance against a squared threshold — int64 only, no `sqrt`; strict `>`, so exactly-at-threshold does not shift
- 5 TEST_CASEs / 12 SECTIONs: construction, bit-exact shift round-trip, threshold boundaries at ±1 mm, squared distance, delegation equality against the free functions

**Key decisions:** service stores state but never decides shift timing —
between-frames enforcement is the caller's job (Phase 10); squared
distance to keep floating point off the exact layer; delegation
wrappers tested for exact equality with the free functions so they
can't drift.

**Deviations:** one auto-fixed compile issue — MSVC C2653 namespace
ambiguity in the delegation tests, resolved with fully qualified
names. Post-phase: the heaviest API churn in the module — later fixes
renamed two methods (`shift_origin` → `set_origin`,
`distance_from_origin` → `distance_squared_from_origin`), deleted
`to_camera_relative` (never read the service's state), fixed the
milestone's only Critical finding (int64 overflow in the threshold
math — flagged by the phase's own threat register, disposition
"accept"), and replaced the bare origin store with a seqlock; an
M0.4-era sweep later corrected the seqlock's writer memory ordering
for weak-memory hardware (2026-06-04).

**Deep-dive:** posted in #technical-devlog

## Phase 10: Rendering Integration — ✅ 2026-04-22

**Goal:** wire `CoordinateService` into the rendering pipeline — the
triangle drawn camera-relative from int64 world positions, coordinate math
visible on screen for the first time.

**What it delivered:**
- `camera.hpp` (15 lines): `struct Camera { coords::Vec3i64 position{}; }` — one field, orientation/fov/matrices deferred until code demands them
- `VulkanContext` ctor-injected with `coords::CoordinateService&` (non-owning); `main.cpp` owns service, camera, and `triangle_world_pos{500, 0, 0}` (0.5 m on +X)
- Single vertex-stage push constant: 16-byte `PushConstants` POD (vec3 + std430 pad, `static_assert`-pinned), `vk::PushConstantRange{eVertex, 0, sizeof(PushConstants)}` in the pipeline layout, written via `cmd.pushConstants<PushConstants>` between `bindPipeline` and `draw`
- `triangle.vert`: `layout(push_constant)` block + `pc.offset.xy * SCALE` with `SCALE = 0.5` (1 m → 0.5 NDC); `triangle.frag` untouched to avoid VUID-vkCmdPushConstants-stageFlags-01795
- `render_frame(camera_pos, object_pos)` converts int64 → float32 inside via `to_camera_relative`; offset captured per command buffer at record time, so both frames in flight hold independent values
- F-key smoke: `SDL_SCANCODE_F` + `!event.key.repeat` guard shifts origin +1 km on X per press with a stdout log line; 16 presses → 16 log lines, triangle stationary

**Key decisions:** push constant over UBO — descriptor-set
infrastructure buys nothing at one object; the renderer borrows
positions per call rather than owning camera/object state;
`shift_origin` stays callable only from `main.cpp` between frames,
never from the render path; no new tests — Phases 7–9's math-lock
covers the arithmetic, proof here is compile-clean + validation-clean
+ visual.

**Deviations:** one formatting-only change (comments reworded to
satisfy literal grep acceptance checks). Post-phase (2026-04-24): the
F-key test got rescoped — `to_camera_relative` never reads the origin,
so the triangle staying put proves the shift path wires up, not
render-path origin invariance; the real test needed a moving camera
(M0.3+). The F-key handler's stdout log line also block-buffers when
piped instead of read from a terminal, hiding the `Origin shifted`
lines until process exit. Debug runs of this phase also surfaced
M0.1's VUID-00067 semaphore bug, fixed in Phase 10.5 before the
milestone closed. Later fixes dropped the `to_camera_relative` service
method Phase 10 called, and renamed `shift_origin` → `set_origin`.

**Deep-dive:** posted in #technical-devlog

## Phase 10.5: Semaphore Reuse Fix — ✅ 2026-04-22 (INSERTED)

Inserted between Phases 10 and 11 by the 2026-04-22 tech-debt
decision — the first debt item closed in its own dedicated phase
before milestone end, with 8 decisions pre-locked going in.

**Goal:** resolve `VUID-vkQueueSubmit-pSignalSemaphores-00067` — a latent
M0.1 frames-in-flight design bug surfaced by Phase 10's debug runs on the
AMD Radeon RX 7900 XT (Vulkan 1.4.344): 14 hits in ~5 s of rendering.

**What it delivered:**
- `render_finished_semaphores_` changed from `std::array<vk::raii::Semaphore, MAX_FRAMES_IN_FLIGHT>` (2 slots, `current_frame_`-indexed) to `std::vector<vk::raii::Semaphore>` sized at runtime to the swap chain image count (3) and indexed by `image_index` from `vkAcquireNextImageKHR` — the signal semaphore is guaranteed unsignaled at submit
- New `create_render_finished_semaphores()` split out of `create_sync_objects()`, called after `create_swap_chain()`; rebuild uses `.clear()` + `emplace_back` (Vulkan-Hpp issue #1218 — `operator=` on live `vk::raii` objects leaks the prior handle)
- `recreate_swap_chain()` rebuilds the vector after `waitIdle()` + `create_swap_chain()`, sized to the new image count, which can change on resize
- `image_available_semaphores_`, `render_fences_`, `MAX_FRAMES_IN_FLIGHT = 2` all untouched — fence-gated per-frame reuse is correct for those
- A recorded decision to take Approach A over `VK_KHR_swapchain_maintenance1`, kept as a rejected-but-recorded future lever
- Stderr VUID scan (2026-04-23: zero hits in 15 s vs 14-hit pre-fix baseline), visual triangle check, MFIF stability — all pass

**Key decisions:** per-image indexing over `VK_KHR_swapchain_maintenance1`
— universal on every Vulkan 1.3 driver, no feature-detection branch, and
the extension's fence-on-present benefits are unused in the current flow;
no `MAX_FRAMES_IN_FLIGHT` bump — the bug is the indexing key, not the
array size; no SemaphorePool abstraction — one consumer; no new Catch2
tests — the 49-test suite pins coordinate math this phase does not
touch, and the automated stderr-scan smoke was scoped to Phase 11.

**Deviations:** none in-phase — 6/6 tasks executed as planned; a scope
guard confirmed exactly 4 files changed. Post-phase (2026-04-25): a
fix switched the method's image-count read to the cached
`swapchain_images_`; the M0.4 orbit demo's `render_orbit_frame` reuses
the per-image pattern; the fix itself has not been reopened since.

**Deep-dive:** posted in #technical-devlog

## Phase 11: Integration Tests + Review Prep — ✅ 2026-04-23

Final M0.2 phase — the milestone's remaining test coverage plus every
artifact its closing review needed. First M0.2 phase with zero manual UAT:
all verification automated (ctest + doc greps).

**Goal:** close M0.2 with the four remaining integration tests and the
review-gate materials it needed; Phase 11 is the last phase before the
milestone closes. 13 decisions were pre-locked going in.

**What it delivered:**
- `test_precision_boundaries.cpp`: one TEST_CASE, 5 SECTIONs at 1 m / 1 km / 1 AU / 1 light-second / 600 AU, every tolerance the true IEEE 754 value for that magnitude; camera-at-distance pattern keeps float32 deltas under the 4 km Debug assert; the 600 AU SECTION locks subtract-before-convert preserving 1 mm exactly above 2^53
- `test_origin_shift_invariance.cpp`: 6 objects × 1000 query steps × 2 sequences, 10^9 mm shift at step 500, 6000 Vec3f64 pairs within `WithinULP(…, 4)`
- `test_no_drift.cpp`: 10M forward+back origin-shift cycles, final origin bit-exact, pure int64 path, 0.27 s in Debug — runs on every PR
- `test_multi_layer_consistency.cpp`: `render_frame` CPU contract (`to_camera_relative`) at 1 AU / 1 light-second / 40 AU; no GPU round-trip
- `tests/data/horizons_earth_j2000.hpp`: Earth J2000.0 state vector as constexpr arrays with the full JPL HORIZONS API query URL, frame, epoch, and units in the header, plus a sanity TEST_CASE
- Three review-gate documents produced for reviewers: a reviewer briefing, a requirement-to-test coverage map spanning all 24 M0.2 requirements, and outreach templates for academic / YouTube-dev / sim-dev reviewer recruitment
- **The Math-Lock rule:** adding tests is always fine; removing tests or weakening tolerances requires written reviewer sign-off
- Suite 49 → 54 TEST_CASEs, full run 0.67 s; 11 requirements closed out

**Key decisions:** storage-only reading of "10M frames" — a full render
loop at that count would take ~46 h; CPU-only for the multi-layer
test — render-to-buffer readback deferred; HORIZONS data committed,
orbit test deferred to M0.3's integrator; three focused review
documents over one wall-of-text briefing; fuzz testing deferred as
out of scope for this milestone; flat test layout kept; a scope
fence — zero `engine/` edits, verified by empty `git diff
--name-only` across all 10 commits; math-lock scoped as a standing
project-wide rule rather than confined to this milestone's review
materials.

**Deviations:** none in-phase — 11/11 tasks as planned; same-day code
review found 2 warnings (ULP-exponent comment, constexpr-tautology
REQUIRE), fixed same-day. Post-phase, the prepared materials fed
directly into the milestone's close-out: a 2026-04-23 pre-peer review
(12 findings, including the 22,000 km DE430/DE441 HORIZONS error), an
expanded property-based suite (54 → 61 tests), and the review-gate fix
series — M0.2 merged 2026-04-25.

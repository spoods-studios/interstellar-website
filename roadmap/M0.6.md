## Phase 30: Zero-Alloc WH Step — ✅ 2026-06-24

**Goal:** eliminate the per-step heap allocation on the WH hot path before
Phase 32's live seed flip magnifies its call frequency — a mechanical,
bit-exact refactor with no semantic or numeric change.

**What it delivered:**
- `wh_step` gains a trailing `std::span<State> helio_scratch` parameter
  (6→7 arg); the per-call `std::vector<State> helio(n)` is replaced by
  `helio_scratch.first(n)`
- `PhysicsWorker::wh_helio_scratch_` — a new member, `assign`ed to
  `body_capacity()` once at the ctor, mirroring the existing `accel_` /
  `ias15_scratch_` reserve-once discipline; production passes
  `.first(active)` of it plus an `active <= capacity` guard
- A size guard (`helio_scratch.size() >= n`, placed after the early
  `dom >= n` no-op return so the OOB-dom test path never trips it) and an
  aliasing assert (`states.data() != helio_scratch.data()`)
- All 30 direct `wh_step` call sites across 11 test files updated to the
  7-arg form; `test_wh_step` two-body golden vectors relock BIT-EXACT —
  35,778 `[wh]` assertions unchanged on Debug + Release, no tolerance
  widening
- New `test_wh_zero_alloc.cpp`: a TU-local `::operator new` counter proves
  a 100-step WH burst allocates zero times once the scratch is pre-sized
- Determinism held throughout: `nbody_force.cpp` byte-unchanged at every
  commit; full suite 405/405 green on both Debug and Release

**Key decisions:** worker-owned scratch reserved once at ctor rather than
resized per call — the same lifecycle as `accel_`/`ias15_scratch_`; the
aliasing assert compiled under the testing macro at landing time (a later
gate fix made it a plain Debug assert); the zero-alloc gate verifies the
already-completed hoist rather than driving new production code, so its
green commit has no separate feature counterpart.

**Deviations:** one auto-fixed blocking issue — `test_det04_cross_platform.cpp`
needed `<vector>` added once its call sites gained a local scratch vector.
Post-phase: Phase 44 (M0.8) added an oblateness-aware `wh_step` overload
that forwards to this 7-arg signature with an empty oblateness table,
carrying `helio_scratch` and its guards unchanged into every later WH
extension.

## Phase 31: Perturbation-Ratio Eligibility Gate — ✅ 2026-06-24

**Goal:** add a dimensionless perturbation-ratio eligibility gate that
catches geometry/spacing-dependent WH invalidity the existing mass-ratio-only
gate misses — a table can be star-dominated by mass yet still carry a bound
satellite pair whose mutual coupling breaks heliocentric WH's small-
interaction premise.

**What it delivered:**
- `wh_perturbation_eligible(span<const State>, BodyTable, dom, mu_dom) ->
  bool` — per-body max-of-ratios, pure `+−×÷`, no `std::sqrt`/libm on the
  path (confirmed only the TU's own `det_sqrt`/`kepler_step` as undefined
  symbols); corrected dimensional formulas found and fixed during
  implementation, not review: `a_dom,i² = mu_dom²/r⁴` (mu_dom already a GM,
  no double-G) and `a_int,ij² = (G·mass(j))²/r_ij⁴` (`r⁴`, not `r⁶`)
- `kPerturbationRatioSqMax = 1.0e-4` (η_max = 0.01, the Hernandez & Dehnen
  2023 / WHFast literature anchor), calibrated against the WH-valid
  barycentric Sun/EMB/Mars/Jupiter config's Brouwer √t slope (measured
  η² = 1.4156e-08, ~3.85 orders below threshold) — **the full shipped
  Sun/Earth/Moon/Mars/Jupiter seed FAILS the gate** at η² = 0.1601
  (η ≈ 0.40, the Earth–Moon pair), correctly: heliocentric WH cannot model
  a bound satellite
- A ctor-only AND wired as a third condition inside the existing
  star-dominated branch of `PhysicsWorker`'s constructor, with a distinct
  fallback log line; `run_due_steps()`'s REGULAR arm verified byte-
  unchanged; ENCOUNTER-classified pairs still route through IAS15
  regardless of the ctor decision (a negative control)
- A dedicated eligibility test (8 cases / 36 assertions) — headline
  bound-Moon rejection using the real J2000 Earth–Moon geocentric offset
  (~402,448 km) from the production seed; a sanctioned migration of 3
  M0.5 WH-selection fixtures whose configs the new gate correctly
  downgrades, re-parameterized (not weakened) to preserve original coverage
- Determinism preserved (`nbody_force.cpp` diff empty every commit);
  Release 416/416, Debug 413/413 (+10 each from the initial baseline)

**Key decisions:** the perturbation-ratio gate as an additional AND on
`kDominantMassRatio`, never a replacement (the mass-ratio constant is its
own math-lock); ctor-only evaluation, per-step hysteresis named as an M1.x
deferral (dynamic regime migration only matters once N grows); calibrate
against the WH-valid barycentric config once the full 5-body seed's
failure was understood as correct rather than a bug — the finding rescoped
the milestone ("Heliocentric WH is invalid for bound satellites"): M0.6
narrowed to Sun/EMB/Mars/Jupiter, satellites split to a new M0.7 built on
this phase's η primitive.

**Deviations:** two dimensional bugs in the η formulas (double-counted G,
wrong power of r) were caught and fixed during implementation, before
calibration — not found later in review. Wiring the gate broke 3
pre-existing M0.5 fixtures; per the ctor-wiring stop constraint this went
to the user, who approved a fixture migration rather than a golden-test
override.

## Phase 32: Live Seed Flip + Documentation — ✅ 2026-06-24

**Goal:** ship Wisdom-Holman (WH) on a real path without touching the
existing 5-body high-fidelity default — Phase 31's calibration found the
full Sun/Earth/Moon/Mars/Jupiter seed fails the perturbation-ratio gate
(Earth-Moon η²≈0.16 ≫ 1e-4), so this phase ships WH by ADD-ALONGSIDE on the
one config where heliocentric WH is valid, plus seats WH in the design
document's warp tier table.

**What it delivered:**
- A second `PhysicsWorker` in `main.cpp` on the Sun/EMB/Mars/Jupiter
  barycentric config (EMB = one body at Earth's heliocentric position,
  mass `M_EARTH + M_MOON`), mirroring an existing energy-test assembly
  verbatim — no new IC literals. `.method = Wh{}`, `.step_dt = 1.0e5` (the
  Brouwer-validated warp step), `.mu_central = 0.0`. The existing 5-body
  `worker_config` is unchanged (`.method = Yoshida4{}`, `step_dt = 300.0`)
- Runtime proof WH genuinely runs, not gate-downgraded: an NDEBUG-stripped
  assert on the selected-method index in `main.cpp` plus a new dedicated
  test case constructing the identical EMB assembly and asserting the
  same — complementing the pre-existing generic 2-body routing pin and the
  bound-Moon rejection pin (unchanged)
- Determinism preserved: the diff on `nbody_force.cpp` is empty — this is a
  `Config`-assembly change, not a kernel change. EMB worker declared after
  the 5-body worker, not wired into `OrbitDemo`/render/warp — unconsumed
  M0.7 scaffolding, no Moon-reattachment logic
- Full suite green: Release 417/417, Debug 414/414; the eligibility-gate
  subset 9/9 (both routing pins); the WH-energy subset 3/3 (Brouwer bound);
  the WH-handoff subset 3/3 (both switch directions)
- The design document's tier table gets a labeled Wisdom-Holman row between
  Tier 2 (Yoshida4) and Tier 3 (Yoshida6+Kepler), plus a paragraph naming
  the gap it fills: the long-horizon warp lever, not a small-step speedup;
  correctly excludes bound satellites (deferred to M0.7)

**Key decisions:** ADD-ALONGSIDE over replace — WH shipped by a second
worker on a valid config rather than forcing WH onto a table it fails on;
reuse the exact validated assembly rather than an unvalidated new
configuration; two independent runtime proofs (assert + test pin) against
the spoofing risk of a gate-downgraded no-op; the documentation edit
confined to the existing four-tier framing, no invented tier names.

**Deviations:** none from the plan. Implementation-detail only: `main.cpp`
writes the EMB mass fully-qualified to match its existing `body_props`
idiom, while the test file uses the bare form via `using` declarations —
both express the same assembly. Post-phase: the design document moved to
the studio vault (2026-06-29); the WH tier row was amended twice more
after M0.7 seated the HJS nested tier alongside it; the EMB scaffolding
worker itself was removed from `main.cpp` on 2026-07-09 (Phase 41) when the
full nested seed became the rendered worker — the eligibility gate it
proved out remains live math-lock infrastructure.

## Phase 33: IAS15 Degrade-Cliff Benchmark — ✅ 2026-06-24

**Goal:** close an M0.5-deferred finding by measuring how whole-system
IAS15 fallback wall-time scales with encounter frequency and N_active on
the now-live WH path, and naming the future MERCURIUS (per-encounter
hybrid) re-add condition — read-only, no engine source change.

**What it delivered:**
- A benchmark-tagged sweep of `N_active ∈ {3,5,10,20} × enc_freq ∈
  {0,1,2,4}` per 100 steps over 2000 WH steps, on a dominant-star fixture
  with a dedicated appended encounter pair (kept separate from the
  wide-orbit satellites so REGULAR-path timing stays clean). Structural
  gates fire regardless of hardware: the dominance-scan guard (avoids an
  M0.4-era dispatch-miss class), a fired-substeps check (IAS15 genuinely
  ran), and a finiteness check.
- The benchmark source was registered in the test build with no default
  test alias — it stays off the default ctest tier.
- The measured results were recorded in the milestone's benchmark record:
  max IAS15-vs-WH ratio **1.76×** (`N=5, enc_freq=1/100`), ratio decaying
  to 1.16× by `N=20`, every window accepting exactly one substep, no sharp
  cliff in the tested grid. The named cost-factor threshold (2.0×) was
  reported not reached. Finding disposition: CLOSED.

**Key decisions:** dominant-star fixture is mandatory (equal-mass fixtures
short-circuit the detector); a dedicated appended encounter pair, not a
perturbed satellite, keeps non-encounter timing uncontaminated; `chrono` +
`WARN` only, never Catch2's `BENCHMARK` macro (its dynamic iteration count
conflicts with the deterministic callsite-counter discipline).

**Deviations:** a dangling `BodyTable` view over a temporary fixture caused
an intermittent assertion failure (2 of 3 runs); fixed by keeping the
fixture alive in a named variable for the whole `N` iteration, folded into
the initial commit before first land.

**Note:** the M0.6 review pass (Phase 35, 2026-06-25) found the "no cliff /
threshold not reached" conclusion above was a small-N, single-substep
artifact — not run-to-run reproducible and measured at a non-shipped step
size. Fixes landed the same day: a default-tier structural-gate test case
and a full rewrite of the benchmark record to an honest "no-cliff floor"
framing with repeated-run spread and an explicit "not characterized"
section. See the deep-dive's drift section for the full account.

**Deep-dive:** posted in #technical-devlog

## Phase 35: Findings + Fixes — ✅ 2026-06-25

**Goal:** close M0.6 through a review pass (2026-06-25) against the
post-Phase-34 tree, every Critical/High finding empirically verified,
every approved finding fixed in-phase as dedicated sub-phases.

**Delivered:**
- **0 Critical, 7 High, 18 Medium, 13 Low**, every Critical/High reproduced
  (0 unreproduced)
- The milestone's central question, found independently across multiple
  review angles: `main.cpp:203`'s second `PhysicsWorker emb_worker`
  (WH-selected) is constructed then discarded — WH runs on no consumed
  trajectory, and its exception channel is never drained, split into a
  dead-error-channel finding (unanimous High) and a scope-honesty
  over-claim in the project's status documentation plus a stale
  unqualified byte-identity claim (unanimous High). Not Critical — the
  physics forcing the EMB-only config (η≈0.40 excludes the Moon) is correct
- Other Highs: the η-eligibility gate summed scalar `Σ|a_ij|²` instead of
  vector `|Σa_ij|²`, falsely rejecting cancelling-perturber configs,
  escalated to High after empirical reproduction confirmed genuine
  cancellation; the IAS15 degrade-cliff benchmark's structural gates are
  hidden from default `ctest` registration; `wh_step`/
  `compute_accelerations` scratch/output preconditions are assert-only,
  ASan-confirmed Release heap/stack OOB under a stripped-assert build; the
  energy-drift alarm plus angular-momentum-deviation oracle have zero test
  coverage; the "no degrade cliff" plus "threshold not reached" benchmark
  conclusions are small-N artifacts, non-reproducible run-to-run (N=3
  straddled the threshold across repeated runs)
- All 7 Highs **APPROVED, fixed in-milestone, no disputes or deferrals**:
  the η gate was rewritten to a true vector-sum norm via `det_sqrt`,
  threshold unchanged, folding 6 Mediums; fail-closed Release guards were
  added ahead of the existing asserts; a non-benchmark-tagged structural
  test case was registered in default `ctest`, folding 3 Mediums;
  value-bearing alarm/angular-momentum-deviation tests plus a `method=`
  telemetry field were added, folding 3 Mediums; both worker error channels
  were drained through a fleet helper plus a poll+alarm, folding 3
  Mediums; the benchmark record was rewritten honestly, the milestone
  status docs were reworded, and the remaining doc Lows were applied
- **GATE CLOSED** 2026-06-25; suite Debug 445/445, Release 449/449;
  `nbody_force.cpp` no longer byte-identical to the M0.5 close (the
  same-day sanctioned strong-type override plus the Release guard),
  locked kernel logic and emitted FP instruction stream both verified
  unchanged
- Merged same day. Two Windows-CI regressions caught at the merge gate and
  fixed before merge rather than deferred: a Vulkan SDK version revert and
  an MSVC POSIX-header portability shim for the alarm test

**Key decisions:** fix all 7 Highs in-milestone rather than deferring
any — the one legitimate deferral surface (the O(N²) ceiling) was already
named with a blocker from the M0.5 close and wasn't reopened here; the
η-gate finding's severity resolved up to High by empirical reproduction of
genuine vector cancellation, not by how many reviews flagged it; the
scope-honesty finding resolved as a wording fix (the physics was already
correct) rather than forcing WH into the consumed render path a milestone
early.

**Deviations:** none against the disposed plan — 7/7 Highs approved and
fixed as scoped, no disputes, no re-deferrals. Post-gate: two Windows-CI
regressions surfaced at the merge gate itself and were fixed same-day
before merging rather than deferred.

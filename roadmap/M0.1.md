# Roadmap Detail — M0.1 Vulkan Bootstrap

> **Granular retroactive roadmap.** One entry per execution phase —
> the phase-level detail layer beneath the milestone-level roadmap.

**Milestone:** M0.1 Vulkan Bootstrap — colored triangle via Vulkan 1.3 dynamic
rendering on the pre-existing VulkanContext (instance, validation, device,
queues), closed with CI.
**Era:** Era 0 (engine foundations).
**Phases:** 6 (numbered 1–6; global continuous numbering starts at 7 with M0.2).

---

## Phase 1: Window + Surface — ✅ 2026-04-08

**Goal:** application opens a resizable window with a valid Vulkan surface;
resize events detected and flagged for the swap chain phase.

**What it delivered:**
- SDL2 window (1280×720, centered, `SDL_WINDOW_VULKAN | RESIZABLE | ALLOW_HIGHDPI`) owned by VulkanContext
- `vk::raii::SurfaceKHR` wrapping `SDL_Vulkan_CreateSurface`'s raw handle
- Instance extensions supplied by SDL's two-phase `SDL_Vulkan_GetInstanceExtensions` query
- Present-queue-family detection (combined graphics+present preferred) + `VK_KHR_swapchain` device extension gate
- `SdlGuard` RAII first-member for SDL_Init/SDL_Quit ordering; `unique_ptr<SDL_Window, WindowDeleter>` for the C handle
- DPI-safe `get_drawable_extent()` clamped to surface capabilities
- SDL event loop in main.cpp replacing `std::cin.get()` — quit/close/resize/minimize handling, `resize_requested` flag for Phase 2

**Key decisions:** a single-class VulkanContext (~350 lines is
manageable) over splitting into several classes; dual resize detection
(SDL events + Vulkan error codes) — AMD/NVIDIA report OUT_OF_DATE
differently, Wayland never does; RAII member order as the
destruction-ordering mechanism.

**Deviation:** vcpkg SDL2 was installed without its `vulkan` feature —
`SDL_CreateWindow(SDL_WINDOW_VULKAN)` failed at smoke test; fixed via
`sdl2[vulkan]` in the manifest, in-commit.

**Deep-dive:** posted in #technical-devlog

## Phase 2: Swap Chain — ✅ 2026-04-09

**Goal:** a swap chain is operational and resilient to window resizing.

**What it delivered:**
- Swap chain creation: `B8G8R8A8_SRGB` + `SRGB_NONLINEAR` (free gamma correction), MAILBOX preferred / FIFO fallback (NVIDIA FIFO input-lag pitfall), `minImageCount + 1` triple buffering with max-clamp
- `UINT32_MAX` currentExtent handling (Wayland case) via Phase 1's DPI-safe drawable extent
- EXCLUSIVE/CONCURRENT sharing mode from Phase 1's queue-family detection
- `oldSwapchain` recreation pattern — one code path for first creation and rebuild (`{nullptr}` member default)
- Image views as owned RAII wrappers over the chain's raw `VkImage`s (ownership asymmetry: chain owns images, we own views)
- `recreate_swap_chain()` (waitIdle → rebuild → clear flag) consuming Phase 1's `resize_requested` flag from the main loop

**Key decisions:** the whole creation config locked at study time from
cross-referenced swap-chain research spanning 14 sources; dynamic
rendering (Vulkan 1.3) keeps recreation to chain + views only — no
framebuffer machinery. `OutOfDateKHR` handling explicitly deferred to
Phase 4 by name.

**Deep-dive:** posted in #technical-devlog

## Phase 3: Rendering Pipeline — ✅ 2026-04-11

**Goal:** a colored triangle renders via Vulkan 1.3 dynamic rendering — no
legacy VkRenderPass objects anywhere.

**What it delivered:**
- GLSL triangle shaders (hardcoded positions via `gl_VertexIndex` — no vertex buffer) + CMake `glslc` build-time SPIR-V compilation with per-file dependency tracking
- Graphics pipeline: empty vertex input, no cull, explicit RGBA write mask, dynamic viewport/scissor, empty-but-mandatory layout, `PipelineRenderingCreateInfo` in `pNext` with `renderPass = VK_NULL_HANDLE`
- Command recording with synchronization2 barriers: UNDEFINED → COLOR_ATTACHMENT_OPTIMAL → PRESENT_SRC_KHR every frame
- `render_frame()`: fence-gated acquire → record → submit → present, single-frame sync (multi-frame explicitly Phase 4), pre-signaled fence + reset-after-acquire deadlock avoidance

**Key decisions:** dynamic rendering as a hard requirement — never carry
legacy render passes; 10 pitfalls pre-catalogued from 15 sources — zero
rediscovered during implementation; scope drawn at pixels-not-
performance with named successor phases.

**Deviations:** the plan omitted `device().waitIdle()` before shutdown;
added in-task. A latent bug also shipped: acquire's
`eErrorOutOfDateKHR` return-code check was dead code (vk-hpp throws
instead) — pre-research knew, the present path got the try/catch,
acquire didn't; it slept until M0.3 Phase 14.5.

**Deep-dive:** posted in #technical-devlog

## Phase 4: Command Buffers + Synchronization — ✅ 2026-04-20

**Goal:** frames are recorded, submitted, and synchronized for multi-frame
rendering — GPU renders frame N while CPU prepares frame N+1. Smallest
diff of the milestone: 27 insertions, though 20 decisions were locked
to get there.

**What it delivered:**
- `MAX_FRAMES_IN_FLIGHT = 2` with `current_frame_` cycling — 2 not 3 because each extra frame in flight adds a frame of input latency
- Per-frame arrays: fences, image-available + render-finished semaphores, command buffers — every frame slot owns a complete resource set
- `render_frame()` converted to pure per-frame indexing; frame 0 waits only on its own prior submission, never on frame 1
- `waitIdle()` kept for recreation/shutdown (rare events, full stall correct); per-frame fences own the hot path

**Key decisions:** the canonical Khronos frames-in-flight pattern,
deliberately; Phase 3's structure made the conversion pure indexing.
The image-in-flight edge case was documented as deferred, with
rationale ("optional for M0.1").

**Deviations:** two compile fixes from vk::raii sharp edges (constexpr
placement before member arrays; `{nullptr, nullptr}` init for deleted
default ctors). Bigger: **the render-finished-semaphore-per-frame
design was proven wrong in M0.2** — VUID-00067; Phase 10.5 re-sized it
per swap-chain image, indexed by `image_index` — the edge case that
had been deferred as optional.

**Deep-dive:** posted in #technical-devlog

## Phase 5: Clean Shutdown + Validation — ✅ 2026-04-13

**Goal:** application shuts down with zero validation errors and no resource
leaks; confidence the bootstrap is solid. This phase made zero code
changes — the expected and achieved outcome.

**What it verified:**
- Full destruction chain audited bottom-up: sync → command pool → pipeline → image views → swap chain → device → surface → instance → window → SDL_Quit; every link enforced by member declaration order
- `grep vkDestroy|vkFree` → comments only; every Vulkan object is `vk::raii::*`
- `device().waitIdle()` before scope exit — the one runtime guard RAII can't provide (GPU may still be using resources)
- One edge case: pending resize at close is skipped, not honored

**Key decisions:** zero errors *and* zero warnings as the bar, root-cause
fixes only; verification treated as a deliverable.

**Deep-dive:** posted in #technical-devlog

## Phase 6: CI Pipeline — ✅ 2026-04-13

**Goal:** automated build and test on every push; M0.1 closes with
verification running on a clean machine, independent of the dev box.

**What it delivered:**
- GitHub Actions workflow: push to `main`/`milestone/*` + PRs to `main` → Vulkan SDK install (cached), vcpkg (commit-pinned), CMake configure, MSVC build, ctest, shader-artifact check, validation-layer availability check
- Shader `.spv` existence asserted independently of build exit status
- No GPU on CI runners means runtime validation stays a local-development responsibility — a documented headless limit

**Key decisions:** vcpkg pinned by commit for reproducibility; milestone
branches in the trigger list (branch-per-milestone workflow pre-wired);
artifact checks over exit-code trust.

**Deviations:** the Vulkan SDK install action failed on first run —
replaced same-day with a direct LunarG installer + `actions/cache`; the
action returned a week later with a corrected tag, the cache stayed.

**Deep-dive:** posted in #technical-devlog

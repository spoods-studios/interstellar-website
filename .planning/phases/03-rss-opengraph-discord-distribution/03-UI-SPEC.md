# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **This spec EXTENDS `02-UI-SPEC.md`, it does not replace it.** Everything about
> the site's visual language — font stack, 18px/1.6 body, `#fefefe`/`#1a1a1a`/
> `#1d5fae` palette, 65ch column, single 640px breakpoint, spacing tokens —
> is already locked there and is **not restated below except where this phase
> adds to it**. Read `02-UI-SPEC.md` first. Nothing here re-opens a Phase 2
> decision.
>
> **Phase 3's on-page visible surface is three text links and zero new CSS
> rules.** The only genuinely new visual artifact is one hand-authored static
> image (the default OpenGraph card, D-49). The highest-leverage surface in
> the phase is *off-site*: the Discord embed card, specified below as a
> first-class surface even though it renders in Discord's client, not ours.
>
> Every field traces to a locked decision in `03-CONTEXT.md` (`D-45`–`D-56`),
> inherits from `02-UI-SPEC.md`, or is marked `[DISCRETION]` where CONTEXT.md's
> "Claude's Discretion" list left it open. `[DISCRETION]` items are resolved
> into concrete defaults here and re-listed at the bottom for the orchestrator
> to surface — this agent has no user channel.

## Color

Palette unchanged from `02-UI-SPEC.md` — `#fefefe` dominant, `#1a1a1a`
secondary, `#1d5fae` accent, light-only (D-22), no destructive color (no
destructive actions exist in this phase).

**Accent reserved-for list — AMENDED.** Phase 2's list stands verbatim; this
phase appends exactly two non-link uses, both enumerated so the list stays
exhaustive and never becomes "all interactive elements":

Inherited from `02-UI-SPEC.md` (unchanged):
- In-body Markdown links, header nav links, prev/next + back-to-archive links,
  breadcrumb links, TOC entry links, archive index post titles

Added this phase:
- **Header "Discord" nav link and footer "RSS" / "Join the Discord" links** —
  ordinary link accent, identical treatment to every existing nav/footer link
  (D-53: "plain text in the site's existing link accent"). This is not a new
  use of the accent; it is the existing use applied to three more links.
- **The 120×4px rule on the default OG card** `[DISCRETION]` — the only
  decorative (non-link) use of the accent anywhere. It appears on an image
  that never renders inside a page, so it cannot be mistaken for an
  interactive affordance on-site. Rationale: a 1200×630 field of `#fefefe`
  carrying two lines of `#1a1a1a` text is legitimately invisible in a Discord
  scroll; a single hairline is the smallest possible intervention that gives
  the card an edge without introducing shape, gradient, or imagery (D-G).
- **`<meta name="theme-color" content="#1d5fae">`** `[DISCRETION]` — Discord is
  the only major consumer that reads `theme-color`, and it uses it to color the
  embed's left border. This is the cheapest brand cue on the phase's
  highest-leverage surface. **Known side effect the orchestrator must surface:**
  `theme-color` also tints the Chrome-on-Android address bar, which is the one
  place a reader sees blue chrome on an otherwise light-only site. Alternative
  if that is unwanted: `#fefefe` (address bar matches the page; Discord's
  embed bar becomes near-invisible) or omit the tag entirely (Discord picks a
  default grey).

Contrast: `#1a1a1a` on `#fefefe` ≈ 16.9:1 and `#1d5fae` on `#fefefe` ≈ 7.2:1 —
both comfortably WCAG AA/AAA. The OG card's 32px and larger type is far above
the large-text threshold.

## Copywriting Contract

| Element | Copy |
|---------|------|
| **Primary CTA (header)** | `Discord` — bare, single word, sits as the **fifth and last** nav item after Roadmap. Word parity with the four existing nav items (Devblog / Technical / How It's Made / Roadmap) is what keeps it from reading as an ad. |
| **Primary CTA (footer)** | `Join the Discord` `[DISCRETION]` — the one place the site actually invites rather than merely links. Verb + noun, no exclamation, no "Come hang out". Register matches VOICE. |

Copyright first (unchanged position, D-18), then the two links in that order (RSS before Discord: D-47's slot was reserved first, and it reads quietest-to-loudest). |
| **Feed channel `<title>`** | `Interstellar Engine — Devblog` `[DISCRETION]` — names the site and scopes the feed to announcements, so a subscriber's reader sidebar says what they actually subscribed to (D-45: the feed is the *archive's* feed). |
| **Feed channel `<description>`** | `Milestone announcements from Interstellar Engine — a space engine built from scratch on real n-body physics.` `[DISCRETION]` — reuses the developer-approved D-16 one-liner rather than inventing a second site description. |
| **Feed `<language>`** | `en` |
| **Per-item title / description** | Post title (D-14/D-33 rules, unchanged) and the D-51 auto-extracted first-prose-paragraph, D-52-truncated to the last complete sentence under ~160 chars with a trailing `…` when the cut lands mid-paragraph. **One extracted value, three consumers** (`<meta name="description">`, OG/Twitter description, RSS item description) — never three separately-authored strings. |
| **`og:site_name`** | `Interstellar Engine` — constant on every page. |
| **`og:title`** | The page's own `<title>` value, unchanged. Do not append `" — Interstellar Engine"` a second time inside `og:title`; `og:site_name` already carries it and Discord renders site name separately, so appending produces a visibly doubled brand in the embed. |
| **`og:image:alt` (default card)** | `Interstellar Engine — a space engine built from scratch on real n-body physics.` |
| **`og:image:alt` (per-post hero, D-48)** | The post's own title. Do not invent descriptive alt text for engine plots the site layer cannot interpret. |
| **404 embed treatment** | Inherits everything: default OG card, existing D-19 copy as title/description source, both CTAs, the `<link rel=alternate>`. No 404-specific copy is added this phase. |
| **Empty state** | Not applicable. The feed cannot legitimately render empty — the `devlog` collection is populated (9 announcements, D-43) and an empty collection is a **build-time error** (D-10/D-33), not a UI state. Do not build a "no posts yet" branch. |
| **Error state (build-time, not rendered UI)** | Three loud failures, all console/CI messages naming the offender, none of which produce client-facing copy: (1) D-54 — invite URL constant unset or still placeholder → build fails **naming the constant**; (2) D-48 — a detected per-post hero image that doesn't resolve → build fails **naming the post and the path**, never silent fallthrough to the default card; (3) D-46 — a feed-content image whose relative path can't be rewritten to an absolute URL. |
| **Destructive actions** | None. No forms, no deletes, no irreversible client action. No confirmation copy needed. |

**Link target behaviour** `[DISCRETION]`: both Discord links open in a new tab
(`target="_blank" rel="noopener noreferrer"`). Rationale: the invite leaves the
site entirely, and the site's whole premise is being the *permanent home* a
reader was mid-way through. This is the site's only external chrome link — the
RSS link stays same-tab (feed readers intercept it, and a raw XML tab is a
recoverable back-button away).

## The Default OpenGraph Card (D-49) — Full Composition Contract

The only new visual artifact in the phase. Authored as SVG in-repo, exported
**once** to PNG, both committed.

| Property | Value |
|----------|-------|
| Source | `src/assets/og-default.svg` — hand-authored, committed, is the editable master |
| Published | `public/og-default.png` — committed binary, served verbatim by Astro from `public/` (same mechanism as `public/favicon.svg`) |
| Dimensions | **1200 × 630** (1.91:1) — the OG convention Discord, X, Slack and iMessage all crop to; anything else gets letterboxed or center-cropped by at least one of them |
| Format | PNG-24, no alpha (the background is opaque `#fefefe`; a transparent card renders black in dark-mode Discord) |
| Target file size | ≤ 200 KB — flat color and text should land far under this; if it doesn't, the export ran at the wrong scale |
| Absolute URL | `https://spoods-studios.github.io/interstellar-website/og-default.png`, built from `site` + `base` (`astro.config.mjs:113-114`), never hardcoded (SITE-02, D-50) |

### Layout

Canvas 1200×630, background `#fefefe` full-bleed. 96px safe inset on all four
sides. All content is a single block, **centered horizontally** and **optically
centered vertically** (block height ≈ 391px → block top ≈ 120px):

```
┌──────────────────────────────────────────────────────┐  #fefefe
│                     ← 96px safe inset →              │
│                                                      │
│                         IE                           │  112px / 600 / #1a1a1a
│                        ↕ 40px                        │
│              Interstellar Engine                     │  84px / 600 / #1a1a1a
│                        ↕ 32px                        │
│                    ▬▬▬▬▬▬▬▬                          │  120 × 4px / #1d5fae
│                        ↕ 32px                        │
│  A space engine built from scratch on real n-body    │  32px / 400 / #1a1a1a
│                     physics.                         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

- **Glyph** — the exact letterform from `public/favicon.svg:3` (`IE`, same
  family, weight 600, `#1a1a1a`). Do **not** reproduce the favicon's `#fefefe`
  background `<rect>` on the card; it is invisible against the card background
  and would only add a stray element.
- **Wordmark** — `Interstellar Engine`, matching `header .wordmark`
  (`global.css:59-64`) in family, weight and color, scaled up.
- **Accent rule** — 120px wide, 4px tall, `#1d5fae`, centered. See the Color
  section's amended reserved-for list.
- **Tagline** — the D-16 one-liner **verbatim**, including the terminal period:
  `A space engine built from scratch on real n-body physics.` Two lines at
  32px within the 1008px content box is expected and fine; center both lines.

### Authoring constraints

- The SVG's `font-family` must be the **literal Phase 2 stack string**
  (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial,
  sans-serif`) so the card is typographically identical to the live header
  wherever it resolves.
- **Rasterize once, commit the PNG.** SVG `<text>` resolves against whatever
  fonts the rendering machine has — this is exactly why D-49 says "exported
  once to PNG": after export the result is deterministic for every viewer.
  Rasterizer choice is the planner's (`rsvg-convert`, `inkscape --export-type=png`,
  or headless Chromium all work) — **no new npm dependency may be added for
  this**; it is a one-time authoring step, not a build step (D-49's whole
  rationale for rejecting satori/resvg/sharp).
- The exported PNG must be **eyeballed once** against the live header before
  commit — if the fallback resolved to something that isn't the site's font,
  re-export on a machine where it does.
- No generative imagery, no gradients, no photography, no engine screenshot
  (D-G, and D-49 explicitly rejected reusing `m0.8-hero-precession.png`).

|---------------|------------|----------|

| Site name (small, above title) | `og:site_name` | `Interstellar Engine` |
| Title (accent-colored, links out) | `og:title` | The page's own title, unduplicated (see Copywriting) |
| Description | `og:description` | D-51/D-52 extracted + truncated value |
| Image | `og:image` | **Large image**, not a thumbnail. Per D-48: real hero (`assets/m0.7-hero-contrast.png` 1900×1060, `assets/m0.8-hero-precession.png` 1920×1080 — both already ≥1200×630 and ~16:9) on those two posts; `og-default.png` everywhere else. |
| Image layout | `twitter:card` | `summary_large_image` `[DISCRETION]`. Required regardless for X/Twitter (DIST-02 names Twitter Card explicitly). Sources conflict on whether Discord reads it; if Discord ignores it, Discord's own og:image sizing still yields the large layout at these dimensions. Small-thumbnail fallback is acceptable, blank is not. |

Also emit `og:type` (`article` on the 9 announcements + 55 deep-dives + 8
roadmap detail pages, `website` on indexes/pages/404) and
`article:published_time` **only where a real date exists** `[DISCRETION]` —
announcements have dates; deep-dives explicitly do not (D-33) and must not
have one fabricated.

**`og:image` and `og:url` must be absolute** (D-50), built from `site` + `base`
via the existing `canonicalUrl` expression (`BaseLayout.astro:11`) and the
normalized `base` (`BaseLayout.astro:15-16`). A relative `og:image` yields a
blank embed in every client — this is the single most likely silent failure in
the phase.

**Verification is a paste, not a grep.** The success criterion ("pasting a live
URL into Discord renders a rich embed with title, description, and image") is
only satisfiable against the deployed site. Build-time greps of `dist/` prove
the tags exist; they do not prove the embed renders. Both are required.

## Placement Contract (the three on-page links)

All changes land in `src/layouts/BaseLayout.astro`. No template outside it is
touched, and no per-page work exists (D-53).

### Header — `BaseLayout.astro:31-36`

`Discord` appended as the **fifth** `<a>` inside the existing `<nav>`, after
Roadmap. No wrapper element, no class, no `aria-label`, no icon, no separator
character. It inherits nav's `text-decoration: none`, 14px/1.4, accent color,
and the existing `flex-wrap` behavior — which is precisely why five items are
safe at any width (`02-UI-SPEC.md`'s wrapping-flex-row decision, no hamburger).

Do **not** mark it as an "active" or "highlighted" nav item; Phase 2 locked
"no separate active-nav color," and D-53 locked no button/pill.

### Footer — `BaseLayout.astro:41-43`

The existing single `<p>` becomes:

```
© 2026 Spoods Studios · RSS · Join the Discord
```

One paragraph, three text nodes and two links, ` · ` separators as literal
text. `RSS` → `{base}rss.xml`; `Join the Discord` → the D-54 invite constant.
Footer links keep the browser-default underline (unlike nav links) — they sit
inline in a sentence-like line where underline does the work of distinguishing
link from copyright text, and no CSS is needed to get it.

### `<head>` — `BaseLayout.astro:20-27`

Joins the existing `<title>` / `description` / `canonical` / `icon` block:
`<link rel="alternate" type="application/rss+xml" title="Interstellar Engine — Devblog" href="{absolute rss.xml URL}">`
(D-47, on every page so reader apps autodetect), the OG/Twitter block, and
`theme-color`.

**CORRECTED 2026-07-22 by `03-RESEARCH.md`:** interpolated `<meta>` attribute
values must use plain Astro `{expr}` interpolation and must **NOT** be passed
through `src/lib/escape-html.ts`. Astro's `{expr}` already escapes attribute
context correctly (including `"` → `&quot;`), whereas `escapeHtml()`
deliberately does not escape quotes and would double-escape `&`. The
WR-01/WR-02 findings applied to `set:html` on titles — a different context
with the opposite requirement. Do not generalize them to attributes.

<objective>
Ship the second slice of the distribution spine: a reader can subscribe to the devblog by RSS,
and every page tells their reader app where the feed is.

The feed carries the full rendered post HTML, produced by the site's own rendering pipeline
so it cannot differ from what the page shows, with image URLs rewritten to absolute so they
actually load in a reader. Discovery lands in the two slots Phase 2 reserved: `<link
rel="alternate">` in the shared `<head>`, and a visible `RSS` link in the footer beside the
copyright and the Discord CTA from Plan 03-01.

Purpose: DIST-01, read literally — one feed, generated from the same collection query as the
archive, so the two structurally cannot drift.
Output: `src/pages/rss.xml.ts` building to `dist/rss.xml`, and feed discovery on all 86 pages.
</objective>

    AstroContainer.create()` outside the loop. For each sorted entry: `const { Content } = await
    render(entry)` then `await container.renderToString(Content)`. This returns byte-identical HTML to

    **Sanitize and absolutize (D-46).** Write a local `absolutize(html, site, entryId)` helper.
    Call `sanitizeHtml` with `allowedTags: sanitizeHtml.defaults.allowedTags.concat(['img'])` — the
    defaults do not include images and would silently delete both hero pictures — and a
    `transformTags` map handling `img` (attribute `src`) and `a` (attribute `href`) with one shared
    transformer. The transformer: when the value begins with `/`, replace it with `new URL(value,
    site).href`; when it is already `http:`/`https:` or a bare fragment, leave it; otherwise **throw**
    naming `entryId` and the offending attribute and value. Do not add `style`, `script`, `iframe`,
    `object`, `embed`, or any event-handler attribute to the allow-list — a monochrome code block in
    a feed reader is graceful degradation, an injection surface is not. Add WHY-comments naming D-46
    and the verified fact that built image URLs arrive root-relative with the base already applied,
    not as the `../assets/…` form the Markdown source shows.

    **Items.** For each entry, in sorted order: `title: entryTitle(entry)`, `pubDate:
    entryDate(entry)`, `link: \`devlog/${entry.id}/\`` (a short relative, resolved against the
    channel `site` below), `description: describeBody(entry.body, entry.id)`, and `content:` the
    absolutized HTML. The package generates `<guid isPermaLink="true">` from `link`, and the
    collection id already carries date and slug, so guids are unique even for two posts sharing a
    title or a date.

    **Channel.** `getRssString({ title: FEED_TITLE, description: FEED_DESCRIPTION, site: new
    URL(base, site).href, items, xmlns: { atom: 'http://www.w3.org/2005/Atom' }, customData:

    type="application/rss+xml"/>\`, trailingSlash: true })`, where `feedUrl` is `new
    URL(\`${base}rss.xml\`, site).href`. Passing `site` **with** the base makes the channel `<link>`
    base-inclusive and lets item links stay short relatives. Do **not** declare the content-module
    namespace in `xmlns` — the package adds it automatically when any item carries `content`, and
    declaring it twice produces a duplicate attribute. Emit the self-referencing atom link from the
    start (D-57): it is the one recommendation the W3C validator predictably raises.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Site-rendered HTML → third-party feed reader | Full post HTML crosses into a client this project does not control and which may render HTML |
| Built asset URL → absolute URL rewrite | The endpoint rewrites attribute values it did not author |

No auth, no session, no user input, no database, no server. Content is git-tracked and
developer-authored via a studio-side promote pipeline; the controls below are defence in depth
against a mistaken or compromised promote, not against an attacker with a form.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02 | Tampering / Elevation of Privilege | `content:encoded` payload in `src/pages/rss.xml.ts` | high | mitigate | `sanitize-html@2.17.6` with an explicit allow-list — package defaults plus `img` only. Inline style, script, iframe, object, embed and event-handler attributes are never added to the allow-list, even to restore code-block coloring; the degradation is monochrome code, which is acceptable |
| T-03-04 | Spoofing | The `transformTags` URL rewriter | medium | mitigate | The rewriter only rebases values beginning with `/` against `context.site`, passes through values already `http:`/`https:` or a bare fragment, and throws naming the entry on anything else — so no attacker-shaped relative path can be silently converted into an off-site absolute URL |
| T-03-06 | Tampering | Channel-level `customData` string interpolation | low | mitigate | The only interpolated values are `FEED_LANGUAGE` (a git-tracked two-letter constant) and `feedUrl` (composed by `new URL` from the config's own site and base); no content-derived value reaches `customData` |
| T-03-07 | Information disclosure | Feed exposing draft content | medium | mitigate | The endpoint calls the same shared visibility guard as every other collection query site, so a `status: draft` post leaves the feed and the archive together; asserted by a trap-and-restore fixture in Plan 03-05 |
| T-03-08 | Denial of service | Unbounded feed growth | low | accept | Uncapped by decision at 9 items; the corpus grows by roughly one announcement per engine milestone, and a static file has no per-request cost |
</threat_model>

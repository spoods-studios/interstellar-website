## Spacing Scale

Declared values (must be multiples of 4), per D-23 ("hand-written minimal CSS
— reading measure, spacing, link accent, responsive rules"):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Meta-line internal gaps, TOC item vertical rhythm |
| sm | 8px | Nav link gaps, breadcrumb separators, code-fence internal padding (inline) |
| md | 16px | Paragraph spacing fallback, list item spacing, card/index-row padding |
| lg | 24px | Header/footer vertical padding, section gaps within a page |
| xl | 32px | Space between page chrome (header/footer) and content column |
| 2xl | 48px | Space above/below the TOC block, milestone-group heading gaps on index pages |
| 3xl | 64px | Top-level page padding on desktop (outside the ~65ch column) |

Exceptions: none. Prose vertical rhythm (paragraph-to-paragraph, heading
margins) inside the untouched Markdown body is **not** re-specified here —
D-14 locks "body renders untouched," so only `<body>`'s browser-default
block-level spacing plus whatever minimal `line-height`/`max-width` rules
apply project-wide (below) touch it. Do not add per-element margin resets
inside the rendered Markdown body.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (prose paragraphs, list items) | 18px | 400 (regular) | 1.6 |
| Label (meta line, breadcrumbs, nav links, TOC entries, footer) | 14px | 400 (regular) | 1.4 |
| Heading (H2/H3 within post body, index milestone-group headings) | 22px | 600 (semibold) | 1.3 |
| Display (post H1 — the Markdown body's own title, D-14) | 30px | 600 (semibold) | 1.2 |

Two weights only: 400 regular (body text, labels, nav) and 600 semibold
(headings, wordmark). No 500, no 700 — matches "boring is correct."

**Primary focal point per template** (the eye's first landing, one per page —
nothing competes with it):
- Reading pages (`devlog/`, `technical/`, `roadmap/`, `pages/`): the Markdown
  body's own H1 at Display 30px. The meta line below it is Label 14px and must
  read as subordinate; the TOC sidebar sits outside the reading column and is
  Label-sized for the same reason.
- Index pages (`/`, `/technical/`, `/technical/m0.X/`, `/roadmap/`): the
  vertical list of post titles. Group headings (Heading 22px) structure the
  list but must not out-weigh it; the homepage one-liner is Body 18px and sits
  above the list as context, not as a hero.
- 404: the "page not found" line at Heading 22px, with the home link directly
  under it.

There is no hero image, no card grid, and no full-bleed element anywhere in
this phase (D-G: no generative imagery; D-29: `hero_visual` ignored in v1) —
the type scale alone carries hierarchy.

Body size note: 18px (not the template's 16px suggestion) is a deliberate
[DISCRETION — CONFIRMED 2026-07-22] call — this is a long-form reading
site (60 technical deep-dives, prose-heavy announcements) at a fixed ~65ch
measure; 18px/1.6 is the standard "comfortable long-form reading" pairing
(matches D-23's reading-measure intent better than the template default).
Reasonable alternative: 16px/1.5 if the developer prefers a denser page.

Do not restyle heading sizes *inside* the rendered Markdown body via
per-tag CSS beyond this single global rule — D-14 forbids body-specific
overrides (no H1 dedup logic, no injected title header). The Heading/Display
rows above describe the *inherited* global H1–H3 styling that applies
uniformly wherever those tags appear (post body headings, TOC-linked
headings, index group headings) — one rule, not a body-specific one.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fefefe` (near-white) | Page background |
| Secondary (30%) | `#1a1a1a` (near-black) | Body text, headings, nav text |
| Accent (10%) | `#1d5fae` (restrained mid blue) | Link color only — see reserved-for list below |
| Destructive | not applicable | No destructive actions in this phase (no forms, no delete/confirm flows) |

Light-only (D-22) — no `prefers-color-scheme` media query, no dark palette,
no CSS custom-property theming layer at all in this phase (SITE-05 is v2;
building even an unused theming hook would be scope creep against "add
nothing between reader and text").

**Accent reserved for** (D-22: "one restrained link accent" — exhaustive list,
never "all interactive elements"):
- In-body Markdown links (prose hyperlinks, resolved wikilinks per D-39)
- Header nav links (Devblog / Technical / How It's Made / Roadmap) — same
  accent, no separate "active nav item" color; current page indicated by
  wordmark-adjacent position only, not a color change
- Prev/next and back-to-archive links (D-15)
- Breadcrumb links (generated cross-links, D-35)
- TOC entry links (D-42)
- Archive index post titles (each entry links to its post)

Accent color exact value `#1d5fae` is [DISCRETION — CONFIRMED 2026-07-22]
— CONTEXT.md explicitly leaves "link accent color" to Claude's judgment. This
is a standard WCAG-AA-passing mid-blue against `#fefefe` (contrast ratio
~7.2:1, comfortably over the 4.5:1 body-text threshold and the 3:1 large-text
threshold) — chosen over a warmer/brand color because no brand palette has
been locked for the website yet and blue-as-link is the most legible,
least-surprising default for a "quiet reading surface." Visited-link state:
not specified — leave `:visited` unstyled (inherits the same accent) to avoid
a second color needing its own reserved-for list.

## Layout & Structural Contract

*(Not a template section, but required to make this UI-SPEC actionable given
the phase's four-content-tree scope — recorded here per "be prescriptive, not
exploratory.")*

### Page shell (every page)
- Header: wordmark "Interstellar Engine" (links `/`) + nav (Devblog /
  Technical / How It's Made / Roadmap), single row, wraps to a second row on
  narrow viewports rather than collapsing into a hamburger menu — D-40/no-JS
  bias means no client-side menu toggle this phase. [DISCRETION — CONFIRMED 2026-07-22
  : a hamburger would need JS or a checkbox-hack; a wrapping flex
  row keeps zero-JS intact and four nav items wrap cleanly at any width.]
- Footer: copyright line ("© 2026 Spoods Studios") + an empty RSS-link slot
  (D-18) rendered as inert text or omitted entirely this phase, populated in
  Phase 3 — do not build a disabled/placeholder RSS icon now (nothing to link
  to yet; avoid dead chrome).
- Content column: max-width ~65ch (D-23), centered, with `xl` (32px) horizontal
  padding on mobile shrinking to 0 extra padding beyond the column itself on
  desktop once the column is centered with margin: auto.

### Responsive breakpoints [DISCRETION — CONFIRMED 2026-07-22]
Single breakpoint, mobile-first, at **640px**:
- Below 640px: nav wraps, TOC (if present) renders inline before the body
  (never sidebar — no room), reading column is fluid `100% - 2×16px` padding.
- 640px and above: reading column is fixed ~65ch, centered; TOC (on technical/
  roadmap pages, D-42) renders as a sticky right-hand sidebar per the decision
  below.
One breakpoint only — this is a single-column reading site with no grid/card
layout that needs intermediate tablet treatment; adding a second breakpoint
would be speculative complexity against "add nothing between reader and text."

### TOC placement and depth (D-42, Claude's discretion)
- **Sticky desktop sidebar** (`position: sticky; top: 32px`) at ≥640px,
  right-hand rail outside the 65ch column (available margin space at desktop
  widths is used for exactly this); **inline block above the body** at
  <640px, collapsible via a native `<details>`/`<summary>` element (zero JS,
  built-in browser disclosure widget — matches the no-client-JS constraint).
- Heading depth: **H2 and H3 only**. Deep-dives and roadmap docs use H2 for
  major sections and H3 for subsections (confirmed in the sampled files);
  going to H4 would surface incidental sub-structure the section index
  doesn't need — matches "the section index the Discord exports already
  carry" (D-42's own stated parity target).
- Applies to `technical/` and `roadmap/` docs unconditionally; per D-42 the
  planner may skip it on short announcement posts — recommend the threshold
  be "render TOC only if the post has 3+ H2 headings," so short announcements
  (many have none) don't get a one-line TOC box.

### Layout factoring (Claude's discretion)
**One shared post-reading layout** (`PostLayout.astro` or similar) parameterized
by: title source (frontmatter vs. body-H1, per D-14/D-33), meta-line content
(per content-tree above), breadcrumb set (per D-35), TOC presence, and
prev/next vs. none. Rationale: all four trees + `pages/` share the identical
visual shell (header, ~65ch column, meta line, optional TOC, footer) — only
the *data feeding* the meta line and breadcrumbs differs per tree. A shared
layout with tree-specific props keeps the "one quiet reading surface" promise
literally true in the CSS/markup, not just in spirit; separate per-collection
layouts would risk the four trees drifting visually over time with no single
place to fix it. Each content tree gets its own **index layout** (archive
homepage, `/technical/`, `/technical/m0.X/`, `/roadmap/`, `/roadmap/m0.X/`)
since index pages differ structurally enough (flat list vs. era/milestone
grouping vs. phase-goal list) that forcing one shared index component would
add more branching than it saves.

### Code blocks (D-41)
- Shiki, Astro's built-in default, **light theme** (matches D-22's light-only
  palette — do not select a dark-optimized theme like `github-dark`).
  Recommend Shiki's `github-light` theme: widely known, high contrast, no
  extra dependency (ships with Astro's Shiki integration).
- Fence padding: `sm` (8px) vertical, `md` (16px) horizontal; monospace stack
  per Design System section above; no line numbers (not in D-41's scope, adds
  visual noise the deep-dives don't need); horizontal scroll (not wrap) for
  long lines — C++ template lines in the sampled files run long, and wrapping
  code breaks column alignment readability worse than a scrollbar does.

### Index/list contract
- **Homepage (`/`, D-13):** flat reverse-chronological list of the 9
  announcements. Each row: title (accent-colored link) + meta line (milestone
  tag + date, D-11). One quiet sentence above the list (D-16).
- **`/technical/`:** grouped by era → milestone (D-40), phase order within
  each milestone (D-34, numeric-aware sort handling decimals like
  `phase-10.5`). Each row: phase number + title, no date (none exists).
- **`/technical/m0.X/`:** same grouping collapsed to one milestone, phase-order
  list.
- **`/roadmap/`:** the transcribed overview (D-37) is prose, not a list — it
  is the `pages/roadmap.md` body rendered through the shared post layout,
  plus a generated list of per-milestone detail links appended as site chrome
  below the prose (outside the Markdown body, matching D-35's "generated,
  never hand-maintained" cross-link rule).
- **`/roadmap/m0.X/`:** phase-by-phase list with goals, sourced from the
  `roadmap/M0.X.md` body (rendered per D-38, own collection) — this page IS
  the Markdown body via the shared post layout, not a separately-templated
  list; the per-phase `Deep-dive:` reference lines resolve to real
  `/technical/...` URLs per D-35/D-39 rather than rendering as inert text.
